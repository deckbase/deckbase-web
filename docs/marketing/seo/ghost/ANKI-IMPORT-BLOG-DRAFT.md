## What to prepare before import

Before importing anything, gather three essentials:

1. **Source backup** - keep original deck exports in a safe location.
2. **Active deck list** - prioritize decks used in the last 30 days.
3. **Template rules** - decide card format standards before bulk import.

If you skip this prep, most migration issues show up as duplicate cards, unclear prompts, and unstable review sessions.

## Import paths: APKG, CSV, spreadsheet workflows

### APKG-style migration

If your workflow relies on native Anki deck packaging, APKG import is usually the fastest route for preserving card bodies and structure.

### CSV migration

For users searching **anki import csv**, CSV is useful when you want explicit field control. It is also easier to batch-clean in external tools.

### Spreadsheet-assisted migration (CSV/Excel)

Spreadsheet cleanup is useful when your legacy cards need normalization (tags, context, duplicates, prompt structure) before import.

## Post-import cleanup priorities

Do cleanup in this order for highest retention impact:

1. **Ambiguous prompts** - rewrite to one concept per card.
2. **Overloaded cards** - split large cards into atomic recall units.
3. **Duplicates** - remove repeated front-side prompts.
4. **Formatting noise** - normalize punctuation and line breaks.
5. **Context tags** - add domain/topic tags where terms collide.

Skipping this order often leads to long sessions and higher lapse rates.

## 14-day migration plan

### Days 1-3: Pilot import

- Import one active deck only
- Resume daily reviews immediately
- Avoid broad cosmetic edits

### Days 4-7: Stabilize

- Track completion rate
- Rewrite top failing cards
- Keep new-card volume conservative

### Days 8-10: Fix recurring defects

- Deduplicate prompts
- Standardize templates
- Clean weak context fields

### Days 11-14: Scale decision

- If completion remains stable and lapses trend down, import next deck batch
- If not, pause imports and continue quality cleanup

This plan minimizes disruption and protects long-term retention during migration.

## Common migration errors and fixes

### Error 1: "Import worked but reviews feel worse"

Cause: card quality drift, not import failure.  
Fix: rewrite weak prompts and enforce one-concept cards.

### Error 2: "Too many duplicate cards"

Cause: legacy overlap across multiple exports.  
Fix: dedupe by normalized front-side text before next batch.

### Error 3: "Session time exploded"

Cause: importing too much at once.  
Fix: reduce new-card intake and scale in smaller deck batches.

### Error 4: "I lost track of what changed"

Cause: no migration log.  
Fix: maintain a simple weekly migration changelog (deck imported, fixes applied, metrics).

## FAQ

### Can I import Anki decks without losing cards?

Yes, if you keep source backups and migrate in stages. Most data-loss concerns come from bulk process errors, not the import path itself.

### Is CSV better than APKG for migration?

CSV is better for field-level cleanup and spreadsheet workflows. APKG is better for preserving packaged deck structure. The best path depends on your cleanup needs.

### How many decks should I migrate at once?

Start with one active deck. Expand only after one to two weeks of stable review metrics.

### Should I tune scheduler settings during migration?

Usually no. Stabilize card quality and completion first. Tuning advanced settings too early often creates noisy outcomes.

## Conclusion

**Anki import** works best as a phased operational process, not a one-click migration event.

Import active decks first, enforce quality gates, and scale only when review metrics stay healthy. That sequence preserves both your card library and your study momentum.
